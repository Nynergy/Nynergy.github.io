It's been a while since the last time I discussed my Linux systems, and there are some changes I want to go over. Let's talk about what those changes are, why they were necessary, and how I went about implementing them without tearing my hair out.

Journey to a New (Old) OS

Those who read my last digital gardening blog post will remember that I was daily driving Gentoo systems, administered with some custom bash scripts. Most of those scripts still exist (I'll touch more on those in a bit), but I've now migrated over to Debian on all my machines.

I don't hate Gentoo, but after compiling my entire system for the nth time in a row, I started feeling a lot more friction than I wanted. Once everything was set up, the system ran perfectly fine, but getting a freshly-installed system to that point was a hassle, especially in a world where I was doing this setup on multiple machines.

I found myself dreading the idea of setting up a new machine, even with my dotfiles repo at the ready. Something needed to change, so I opted to move to Debian for faster installation of a minimal system.

I also felt that my dotfiles were lacking in a few departments, so I added some scripts, pruned some old unused files, and refined what was already there to make the post-install setup as seamless and easy as possible. We'll talk about those refinements later, but first, let's talk about what my post-install process is at this point.

Building the House

My post-install process can generally be broken down into the following steps. This is what I do right after I've rebooted into a freshly-installed system.

Step 1 Bootstrap user
Step 2 Install dotfiles
Step 3 Start up an X session
Step 4 Machine-specific configurations

Bootstrapping the user is very simple; As root, I install sudo and add my personal user to the sudo group. Once I log out and back in again, I can begin the real setup by pulling down my dotfiles.

Installing the Dotfiles

In order to install my dotfiles, I first need to install git and clone the repo into a hidden directory. 

$ sudo apt install git
...
$ git clone https://github.com/Nynergy/dotfiles.git .dotfiles

Once the repo is cloned, we can move into it and run the included install script. 

$ cd .dotfiles/
$ ./install/proliferate

The install script proliferate will look through all the subdirectories in the repo for various property files and perform actions based on the contents of those files. When it finds a links.prop file, it reads source-destination pairs from it and symlinks the source to the destination on my system. When it finds an env.prop file, it reads script-variable pairs and adds them to an environment file.

Some scripts that use this environment file need to be run as root, so I also manually symlink it to a location that root can see it. 

$ cd /
$ sudo ln -s $HOME/.env.sh /.env.sh

Then I can go in and edit the environment file (after installing vim) to populate all the variables.

Installing the Other Stuff

My dotfiles are littered with README.md files that contain lists of dependencies and suggested installs. These usually look something like this: 

#### Dependencies

- Apt
  - tmux
  - rxvt-unicode
  - suckless-tools

#### Suggested Installs

- Apt
  - fonts-terminus

Instead of reading through all these files manually, I also include a multi-stage shell command that will scrape the install targets from all the files and present them on the command line. Usually, I will redirect this list into a temporary file that I can reference while installing packages. 

$ grep -r "\- " ./**/README.md | grep -Ev "Apt|Repos|#|\`" | cut -d '-' -f 2- | sort | uniq | sed 's/ //'

This is a bit of a mouthful, so let's go over what it's doing step-by-step.

grep -r "\- " ./**/README.md Find all lines in any README.md file that contain '- '
grep -Ev "Apt|Repos|#|\`" Remove lines containing stuff we don't care about
cut -d '-' -f 2- Grab only the text after the hyphen delimiter
sort Sort the output lexicographically
uniq Remove duplicate lines
sed 's/ //' Remove spaces

What you're left with in the output is a sorted list of package atoms for apt or links to third-party repositories. Once all of those are installed, I'm ready to start an X session via startx and finish setting up.

Final Configurations

Once inside the X session, it becomes immediately apparent that there is no wallpaper, so I usually install GIMP and create a "blank" wallpaper of the appropriate resolution. This is just a solid color that matches whatever the background of my system color scheme is.

Then I take care of any machine-specific configuration changes. For example, my polybar config file expects the network interface to have a certain name, but it may have a different name on this machine. Another common one is a screen layout file for use with xrandr via my mosaic script.

Speaking of scripts, let's talk about how I've refined my system scripts, and added some new ones to improve my daily driving experience.

Tilling the Land

My dotfiles also contain a bunch of bash scripts that I symlink into my $PATH for ease of use. These system scripts have worked well enough up to now, but there were definitely pain points I wanted to address with some updates. Notably, the ways in which I handled screen layouts and system color schemes was all over the place, with weird quirks, redundant files, and tedious trial-and-error changes.

With this latest round of refactors and refinements, I wanted to overhaul my color scheme management and screen layout selection, as well as introduce some scripts centered around the following ideals:

Reduce Friction Remove Redundancy
Improve Maintenance Increase Quality-of-Life

Let's talk about these ideals and the scripts/changes that embody them.

Reduce Friction

A lot of these scripts have been refined in a way that makes everyday tasks easier, but lintel The script handles opening terminal windows, so the name comes from a window-related architectural term . is the one that I feel most directly reduces friction.

It's a deceptively simple script. All it does is grab a list of project directories from inside a specific folder and present the user with a dmenu selector for those projects. When the user selects one, it opens a new terminal instance, changes to that project's directory, and starts a tmux session.

This may seem kind of inconsequential, but it makes the process of working on programming projects so much easier. Instead of manually navigating to wherever my code base is and getting myself set up to code, I can tie lintel to a system hotkey and jump in right away. Plus, once all tmux panes are closed, the terminal disappears with it, eliminating some cleanup that used to be manual as well.

Remove Redundancy

I used to have two scripts -- doublescreen and triplescreen -- that did the same thing. One was for my 2-monitor setup at home, and another for my 3-monitor setup at work. They used a set of hard-coded display parameters in conjunction with xrandr to set the screen layout of the X session.

Not only was there a lot of code duplication across the two scripts, but if I ever needed to create a screen layout for a new setup, it would require me to create a whole new script. The only things that really differed between the two were the screen configuration details, and the amount of screens. Surely we could pull those out into some kind of configuration file and then use separate files for different layouts?

So that's what I did. I combined the two previous scripts into one called mosaic that reads a layout configuration file provided by the user via dmenu and sets the layout accordingly. You can see one of these files below, describing a setup with three screens. 

output DP-1-3
primary
mode 1920x1080
pos 1920x0
rotate normal

output eDP-1
mode 1920x1080
pos 0x0
rotate normal

output DP-3
mode 1920x1080
pos 4000x0
rotate left

The values are the same ones that xrandr expects, but without any of the leading dashes.

By handling layouts this way, we can now add a new layout by creating another configuration file in the directory mosaic looks in. We've also made it so that only a single script handles setting the layout, not an ever-expanding number.

Improve Maintenance

Maintaining these scripts, as well as any number of other programming projects, can be tedious. Often I will have an idea of how to improve or add to something, but that idea will sit there in some TODO comment for a long time. Upon coming back to it, I may not remember what I meant by it, or I may lack some context for why it was necessary. I also don't like littering my code with a bunch of comments that aren't really documenting anything related to the code itself, but instead what I would like to do to the code.

I implemented isli Pronounced like "eye-lee". I know, taking a lot of creative liberties with this one. to add more context to different maintenance tasks, and to eliminate these vestigial comments in my code. The script is highly inspired by Tsoding's tatr tool, though it has a fraction of the features. In fact, the name is even inspired by it; tatr is a task tracker, while isli is an issue lister.

Very simply, isli looks for a tasks/ directory in the current directory, and looks at the files therein. Inside that directory are a bunch of date-stamped Markdown files containing the name of the issue, its creation date, its status, and some arbitrary information about the issue. See an example of the issue I made for writing this blog post below.

You can use isli to list issues, open existing issues in your editor, and create new issues. At time of writing, I haven't implemented any kind of tag system, query language, issue deletion, or anything like that. Just something simple that makes creating and tracking tasks easy.

Increase Quality-of-Life

One real pain point I had with the previous iteration of dotfiles was the way I handled system color schemes. Before, I would edit the color values in my .Xresources file, reload my terminal settings, view the changes in a new terminal window, and repeat that process until I found colors I liked. Then I would run another script called palette to chop off the head of my .Xresources and dump it in a new color scheme file. Then, I could use another set of scripts to actually select and apply that color scheme to all manner of configuration files across my system.

Not only was this process incredibly tedious, it also meant that my .Xresources file needed to stay in a certain format/order, and my color scheme files needed to obey that same order. This was a problem, so I decided to go a little overboard and make another one of my little TUI projects.

I named the resulting tool rimeglass in reference to Atrus' crystal experiments on Rime in the Myst series. It reads color scheme files from a specific directory and allows the user to scroll through them, updating the terminal colors in real-time as you scroll from scheme to scheme. It also allows the user to edit the current palette of colors and save that palette to a new color scheme file.

With color schemes now decoupled from the .Xresources file, I could finally add some metadata information to it, reorganize my color scheme collection, and eliminate the old palette script. Not only does rimeglass improve my quality-of-life as a Linux user, it also reduced the friction of handling system colors, removed some redundant scripts, and improved the way I maintain my color schemes and associated scripts. I also think it's the best-looking TUI I've made to date. Just don't try to resize the terminal window... @_@
